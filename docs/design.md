## Eng Review Refinements (2026-05-18)

Plan refined through /plan-eng-review. Outside voice (independent cold read) surfaced
6 challenges. After cross-model tension review, the user kept GitHub Pages + 3 templates
(original plan) and accepted these refinements:

- **Skill is standalone.** No gstack admin gates, no preamble bash, no decision-brief
  AskUserQuestions, no telemetry, no capture-learnings section. The skill is PM-facing
  product, not gstack ecosystem citizen. Eliminates 5-10 minutes of admin friction
  before the interview starts.
- **No gstack dependency.** The skill runs without gstack installed. Just Claude Code,
  git, optionally `gh` CLI. The PM (and downstream PMs they share with) do not need to
  install gstack.
- **spec.md + recap.md gitignored by default.** Each template's `.gitignore` includes
  `spec.md` and `recap.md`. The PM keeps them locally; they don't ship to GitHub. The
  recap explains how to delete the .gitignore lines to opt-in to publishing. Protects
  PMs against accidentally publishing employer names, customer names, internal product
  strategy. Privacy by default.
- **URL verification is manual paste-back.** At Min 27, skill prints the github.io URL
  and tells the PM: "Refresh in 1-5 min. Paste here when you see your tool load." No
  Bash curl loop. No token waste. Educates the PM about deployment latency.
- **Classifier eval suite deferred to V2.** First 3 DM'd PMs are V1's classifier QA.
  Risk accepted: if 2+ get bad classifier picks, fix the prompt mid-week.
- **GitHub friction is the learning, not the bug.** Outside voice flagged: non-technical
  PMs may lack git/PAT, Pages toggle is multi-step, p50 may be 60-75 min not 30. We are
  embracing these as part of the "ready for next build" goal — the recap names what the
  PM just learned about git, GitHub, and deployment. See "V1 Failure Modes" below.

## V1 Failure Modes (planned UX handlers)

- **PM has no GitHub account:** Skill detects, prints signup link, waits for PM to
  return with a username. Adds 5-10 min to flow; flag in opening utterance ("if you
  don't yet have GitHub, this adds 5 min").
- **PM has no `gh` CLI authed:** Skill detects, falls back to web UI: "Open
  github.com/new in your browser, name the repo `<suggested-name>`, paste back the URL."
  Adds 2-3 min.
- **Repo name conflict:** Skill suggests `<name>-v2`; if PM picks differently, accepts.
- **Pages toggle confusion:** Skill prints **exact step-by-step instructions**: "1.
  Click Settings. 2. Click Pages in left sidebar. 3. Source = Deploy from a branch. 4.
  Branch = main. 5. Folder = / (root). 6. Save. 7. Refresh page; you'll see a yellow
  notice 'Your site is being built'." This is the highest-failure-rate step; bullet it
  to death.
- **Pages takes > 5 min to propagate:** Skill says "still propagating, keep refreshing
  every 30s; sometimes new Pages take longer on first deploy." Don't time out.
- **Org-owned account with Pages disabled at org level:** Skill detects via Pages 404
  after 5+ min. Recommends PM create a personal repo instead.
- **One-word interview answers:** Skill asks one short follow-up ("Can you say a bit
  more about that?"); accepts the second answer regardless. Prevents 4x follow-up loops.
- **Realistic time budget:** Opening utterance updated: "30 minutes if you have GitHub
  + gh CLI set up. 45-60 if you need to set those up first." Honest expectation-setting
  > broken promise.

Mode: Builder

## Problem Statement

Non-technical product managers who have just installed Claude Code (or are about to) face a "cold start" gap: they know the tool is powerful, but they have never successfully shipped anything with it, and the blank-prompt moment is intimidating. They drop off before the aha moment.

The builder (a PM) had a "major" aha moment when they figured out how to vibe-code efficiently with Claude Code. They want to detonate that same aha moment for other non-technical PMs, on a controlled schedule (30 minutes), with a take-home artifact the PM can show their team — and with enough explicit learning baked in that PM #2's *next* build doesn't require the rails.

## What Makes This Cool

- **The user IS the product.** The builder is a non-technical PM building for other non-technical PMs. No empathy gap.
- **Transmit a feeling, not a tool.** The success metric isn't "tools shipped" — it's "did the PM text a friend within 24 hours." Every design decision is judged against: does this make the aha bigger or smaller?
- **Interview the PM like a PM interviews a customer.** Open with 7 PM-shaped questions instead of a blank prompt. PMs already know how to answer this style of question — it lowers cognitive load and produces a real spec.
- **Show-and-tell as the engine.** The 30-min experience ends with a deployed URL — something a PM posts in Slack and a teammate replies "wait, you built that?"
- **Learning is part of V1.** The PM doesn't just walk away with a thing — they walk away with a recap memo that captures the spec, the prompts used, and what to try next, so their second build doesn't depend on the rails.

## Constraints

- Builder is solo; ship-ability matters more than scope.
- V1 must be ship-able in one weekend (Saturday build, Sunday post).
- V1 cannot install Claude Code on the user's behalf (that's a separate, larger problem). Pre-install onramp ships as a simple link/instruction in the README.
- **All deploy targets must work with zero auth** — GitHub Pages "enable Pages" toggle only, no API tokens, no CLI logins.
- **All template stacks must be debuggable by a non-technical PM** — static HTML/JS + browser-only storage (localStorage), no databases, no server-side code, no build steps the PM has to understand.
- Force-fit any PM idea into one of the 3 V1 templates — never dead-end into a blank prompt.

## Premises

1. **"Cold start" = post-install first-success.** The product targets PMs who already have Claude Code installed (or can follow a one-line link to install it). It does NOT solve install/awareness; it solves "now what?"
2. **30 minutes is the right time-box.** Long enough to ship something real; short enough for a lunch break. Enforced via explicit minute-markers in the skill (see Recommended Approach).
3. **Output is a show-and-tell-able artifact** — deployed URL for web templates, a Loom/screenshot for non-web — that the PM can post in Slack. Not a certificate, not a tutorial completion.
4. **V1 ships as a Claude Code skill** (`/first-build`) installed via `git clone + setup` on the user's machine. Distribution = the gstack/Claude Code skill ecosystem + word-of-mouth screenshots. No skill marketplace dependency. No CI/CD pipeline in V1.

The aha isn't "I built a thing" — it's "a senior engineer just shadowed me for 30 minutes and read my mind." Held as V2/V3 ambition; outside V1 scope.
- **Key signal in the builder's words:** *"The aha moment for me was major."* Not "useful," not "productive" — **"major."** They're not building a tool, they're transmitting a feeling. Reframes the success metric to "did the PM text a friend within 24 hours."

- **Premise tension surfaced:** The subagent's "interview-first" reframe pushed the UX from "one sentence → build" to "7 questions → spec → build." Adopted into the V1 design.

## Approaches Considered

### Approach A: MVP — one skill, opinionated rails, recap exit (CHOSEN, revised post-review to 3 templates)

- One skill `/first-build` with 7 locked interview questions, then build.
- **3 force-fit templates** (revised from 5): internal-tool, dashboard, prototype — all static HTML/JS + localStorage, GitHub Pages deploy.
- Exits with a 1-page `recap.md` capturing the extracted spec, the prompts used, the template chosen, the bending instructions applied, and what to try next.
- One-command "deploy" = enable GitHub Pages toggle on the PM's forked repo; no tokens, no auth.
- Pre-install onramp in README: one-liner pointing at Anthropic's Claude Code install docs.
- **Effort:** S (1 weekend). **Risk:** Low.

### Approach B: LADDERED — two skills, structural learning ladder

- Approach A's `/first-build` plus a sister `/next-build` skill that is intentionally less hand-holdy.
- Learning is structural (two-skill ladder), not just documented (recap).
- **Effort:** M (~2 weekends). **Risk:** Med — `/next-build` design is fuzzy until real PMs have hit V1.
- Deferred to V2 once V1 ships and 3+ PMs report what they did next.

### Approach C: GLASS BOX — one skill, fully transparent and exportable

- Same scope as A, but the skill's own prompts are readable/editable mid-build and the exit artifact is a forkable Markdown "build memo" (spec + prompts + decisions).
- Philosophy: "glass box not magic" — the PM leaves with a reusable spec/prompt library.
- **Effort:** S–M. **Risk:** Low–Med (tension: showing prompts might break the magic).
- Held as a V2 reframe if MVP feedback shows PMs want more transparency.

## Recommended Approach: Approach A (revised)

### The 30 minutes, in explicit minute-markers

- **Min 0:** Skill opens with the verbatim utterance: *"Hi — I'm going to ask you 7 PM-style questions about what you want to build. There are no wrong answers. In about 30 minutes, you'll have a live URL you can share. Ready?"*
- **Min 0–5: Interview.** Skill asks the 7 locked questions in order. PM answers in chat. Skill writes answers into `spec.md` using the schema below as it goes.
- **Min 5–7: GitHub repo precheck.** Skill checks: does the PM have a GitHub account? Does `gh` CLI exist locally and is it authed? If yes, skill runs `gh repo create <name> --public --clone`. If no, skill prints step-by-step instructions for github.com "New repo" and asks the PM to paste the resulting URL back. (Precheck failure modes: no account → link to github.com/signup + wait; `gh` not installed → fall back to web UI; repo name conflict → suggest `<name>-v2`.)
- **Min 7–10: Template selection + force-fit.** Skill runs the classifier (see `template-picker.md`), picks one of the 3 templates, shows the PM the pick + bending instructions verbatim: *"Closest = dashboard, because you described data and a stakeholder audience. Here's how I'll bend it: [3 sentences]. Say 'override' if you want a different template, or 'go' to proceed."* If PM overrides, skill rewrites the classification line in `spec.md`, re-classifies to the chosen template, then says verbatim: *"Got it — switching to <template>. New bending: [3 sentences]. Say 'go' to proceed."*
- **Min 10–25: Build loop.** Skill forks the chosen template into the PM's repo by copying the pre-baked template directory and applying bending. Bending is **config-swap, not codegen** wherever possible (see "Pre-baked bending surfaces" below). Skill asks ~2 clarifying questions inline only when blocked. PM watches files change in their editor.
- **Min 25–28: Push + enable Pages.** Skill runs `git push`, then walks the PM through the 7-step Pages-toggle bullet list (see SKILL.md → "The 7-step Pages-toggle instructions"). The skill cannot click the toggle for the PM — that click stays in the PM's hands, on purpose, as part of the "comfortable with GitHub" learning goal.
- **Min 28–30:** Recap handoff (manual paste-back). Once the PM confirms the "Your site is being built" yellow notice, the skill prints the URL and says: *"Your URL: `https://<username>.github.io/<repo>/`. GitHub Pages takes 1–5 minutes on a new repo — sometimes longer. Refresh that URL every 30–60 seconds. When you see your tool load, paste 'live' here and I'll finish the recap."* While the PM waits, the skill writes `recap.md` (using `recap.md.tmpl`). **No curl polling, no auto-detect** — manual paste-back is intentional: it teaches the PM what deploy latency feels like, keeps the token budget honest, and avoids brittle background processes. **V1 ships with NO telemetry link** — `recap.md.tmpl` includes the reflective question *"Did you text a friend within 24h?"* as prose only, no form. (Google Form added in V2 if the question is worth measuring.)

### The 7 interview questions (locked, V1) + spec.md schema

Each question maps 1:1 to a field in `spec.md`. Skill writes the file as the PM answers.

| # | Question | spec.md field |
|---|----------|---------------|
| 1 | What problem is this for? (1 sentence) | `problem` |
| 2 | Who is this for? Be specific — a person, not a role. ("Sarah on my analytics team," not "analysts") | `persona` |
| 3 | Walk me through their Monday morning — when does this problem show up? | `scenario` |
| 4 | What data or input does this person need to see or work with? | `inputs` |
| 5 | What action should this thing let them take, or what answer should it give them? | `outputs` |
| 6 | How would they know it worked? What's the moment that makes them say "this saved me time"? | `success` |
| 7 | Show me a tool or feature you like — what about it works for you? | `taste` |

The full `spec.md` ends with two appended fields after force-fit:
- `template` (one of: internal-tool, dashboard, prototype)
- `bending` (1–3 sentence prose describing the adaptation)

### The 3 V1 templates (locked) + pre-baked bending surfaces

Each template is a working, deployable static site. Bending is **config-swap**, not codegen, wherever possible — to keep Min 10–25 inside its time budget.

1. **internal-tool** — Static HTML form + localStorage + table view. PM-friendly CRUD for personal lists/trackers.
   - Bending surfaces (config-swap): `fields.json` (field name, label, type), `copy.json` (page title, button labels), `theme.css` (one accent color variable).
   - Codegen only if PM needs a non-standard field type (e.g., file upload).
2. **dashboard** — Static HTML + Chart.js + a hardcoded `data.json`.
   - Bending surfaces (config-swap): `data.json` (pre-baked with 3 example shapes the classifier can pick from: time-series, categorical-bar, two-column-table), `chart-config.json` (chart type, axis labels, title), `narrative.md` (the "what this shows" copy above the chart).
   - Codegen only if PM needs a chart type outside Chart.js's defaults.
3. **prototype** — Static multi-page clickable HTML mockup, no backend.
   - Bending surfaces (config-swap): `pages.json` (page count + nav structure), `copy/*.md` (one Markdown file per page, rendered at build).
   - Codegen only for non-standard interactions (modal, drag-and-drop).

All three: zero build step, zero install, zero runtime dependency beyond a browser. Deploy = `git push` + enable GitHub Pages toggle (PM clicks; skill cannot click for them).

### The force-fit mechanism (defined, not a vibe)

The skill includes a `template-picker.md` with an LLM classifier prompt:

> "Given this PM's spec (problem, user, scenario, data, action, success, taste), classify into ONE of: internal-tool, dashboard, prototype. Then write the bending instructions: 1-3 sentences explaining how to adapt the chosen template to this specific spec. If no template fits cleanly, pick the closest and explain the bend honestly."

There is no "no template fits" branch in V1. The classifier always picks one and always explains the bend. The PM can override the pick verbally before the build starts.

### Non-web artifacts: deferred to V2

All 3 V1 templates are web-deployable, so every V1 build produces a URL. Survey and automation templates (which would need Loom/screenshot artifacts) are deferred to V2. If a PM's spec genuinely can't be web-deployed via the 3 V1 templates, the force-fit picks the closest and adapts — they still get a URL.

## Open Questions

- **Deploy timing UX:** GitHub Pages takes 1-2 minutes to propagate after enabling. The 30-min flow needs to handle that wait gracefully — likely by writing the recap during the propagation wait.
- **Spec.md format:** Should the spec be a structured Markdown the PM can reuse in their next build, or a free-form summary? Leaning structured — feeds the "ready for next build" goal.
- **Bending instructions visibility:** Should the PM see the bending instructions before the build, or only after? Leaning before (transparency builds trust, sets expectations).
- **Recap re-use test:** What's the simplest way to validate that the recap is good enough for PM #2 to attempt build #2 without `/first-build`? Likely: ask one of the first 3 DM'd PMs to try.

## Success Criteria

- A non-technical PM with Claude Code installed runs `/first-build`, answers the 7 interview questions, and within 30 minutes has a live URL on github.io.
- The PM posts a screenshot of their result to Slack/Twitter/LinkedIn within 24 hours of finishing.
- At least 3 PMs DM the builder within 7 days of launch about their build.
- **At least 1 of those DM'd PMs attempts a second build using only the recap (no `/first-build`) and reports back whether the recap was sufficient.** This is the V1-validates-V2-direction signal.
- Self-report: the recap includes an optional 1-question link ("did you text a friend within 24h?") to measure the core metric without invasive telemetry.

## Distribution Plan

- **Pre-install onramp:** README in the GitHub repo with a one-line install instruction pointing at Anthropic's Claude Code docs, plus a tweet-shaped launch post the builder can share.
- **Post-install distribution:** Ship as a Claude Code skill installable via `git clone <repo> ~/.claude/skills/first-build && cd ~/.claude/skills/first-build && ./setup` (same pattern gstack uses). Word-of-mouth via PM Slack communities (Mind the Product, Lenny's, Reforge alums).
- **The artifact IS the marketing.** Every deployed result from V1 includes a footer: *"Built in 30 min with /first-build"* linking back to the repo. Each shipped tool becomes a distribution surface.
- **No CI/CD in V1** — install is `git clone`, no publish step, no GitHub Actions. Add CI when there's a publish pipeline to gate.
